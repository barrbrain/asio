Tutorial Timer.5 - Synchronising handlers in multithreaded programs

This tutorial demonstrates the use of the asio::locking_dispatcher class to synchronise callback handlers in a multithreaded program.

The previous four tutorials avoided the issue of handler synchronisation by calling the asio::demuxer::run() function from one thread only. As you already know, the asio library provides a guarantee that callback handlers will only be called from threads that are currently calling asio::demuxer::run(). Consequently, calling asio::demuxer::run() from only one thread ensures that callback handlers cannot run concurrently.

The single threaded approach is usually the best place to start when developing applications using asio. The downside is the limitations it places on programs, particularly servers, including:

  • Poor responsiveness when handlers can take a long time to complete.
  • An inability to scale on multiprocessor systems.

If you find yourself running into these limitations, an alternative approach is to have a pool of threads calling asio::demuxer::run(). However, as this allows handlers to execute concurrently, we need a method of synchronisation when handlers might be accessing a shared, thread-unsafe resource.

#include <iostream>
#include "boost/bind.hpp"
#include "asio.hpp"

Step 1. We start by defining a class called printer, similar to the class in the previous tutorial. This class will extend the previous tutorial by running two timers in parallel.

class printer
{
public:

Step 2. In addition to initialising a pair of asio::timer members, the constructor initialises the dispatcher_ member, an object of type asio::locking_dispatcher.

An asio::locking_dispatcher guarantees that, for those handlers that are dispatched through it, an executing handler will be allowed to complete before the next one is started. This is guaranteed irrespective of the number of threads that are calling asio::demuxer::run(). Of course, the handlers may still execute concurrently with other handlers that were not dispatched through an asio::locking_dispatcher, or were dispatched through a different asio::locking_dispatcher object.

  printer(asio::demuxer& d)
    : dispatcher_(d),
      timer1_(d, asio::time::now() + 1),
      timer2_(d, asio::time::now() + 1),
      count_(0)
  {

Step 3. When initiating the asynchronous operations, each callback handler is "wrapped" using the asio::locking_dispatcher object. The asio::locking_dispatcher::wrap() function returns a new handler that automatically dispatches its contained handler through the asio::locking_dispatcher object. By wrapping the handlers using the same asio::locking_dispatcher, we are ensuring that they cannot execute concurrently.

    timer1_.async_wait(dispatcher_.wrap(boost::bind(&printer::print1, this)));
    timer2_.async_wait(dispatcher_.wrap(boost::bind(&printer::print2, this)));
  }

  ~printer()
  {
    std::cout << "Final count is " << count_ << "\n";
  }

Step 4. In a multithreaded program, the handlers for asynchronous operations should be synchronised if they access shared resources. In this tutorial, the shared resources used by the handlers (print1 and print2) are std::cout and the count_ data member.

  void print1()
  {
    if (count_ < 10)
    {
      std::cout << "Timer 1: " << count_ << "\n";
      ++count_;

      timer1_.expiry(timer1_.expiry() + 1);
      timer1_.async_wait(dispatcher_.wrap(boost::bind(&printer::print1, this)));
    }
  }

  void print2()
  {
    if (count_ < 10)
    {
      std::cout << "Timer 2: " << count_ << "\n";
      ++count_;

      timer2_.expiry(timer2_.expiry() + 1),
      timer2_.async_wait(dispatcher_.wrap(boost::bind(&printer::print2, this)));
    }
  }

private:
  asio::locking_dispatcher dispatcher_;
  asio::timer timer1_;
  asio::timer timer2_;
  int count_;
};

Step 5. The main function now causes asio::demuxer::run() to be called from two threads: the main thread and one additional thread. This is accomplished using an asio::thread object.

Just as it would with a call from a single thread, concurrent calls to asio::demuxer::run() will continue to execute while there is "work" left to do. The background thread will not exit until all asynchronous operations have completed.

int main()
{
  asio::demuxer d;
  printer p(d);
  asio::thread t(boost::bind(&asio::demuxer::run, &d));
  d.run();
  t.join();

  return 0;
}
